     冲突。 → 全小写 trace/debug/info/warn/error/critical/off (推荐)
     · default_formatter 的输出格式怎么定？ → 固定格式：[time] [level] [name] [pid:tid] message (file:line) (推荐)
     · LogSink 抽象基类要不要带 sink-level 过滤？spdlog 的 sink 自己有 level，可以独立于 logger 关掉某些 sink
     的低等级输出。 → 不带，过滤完全由 LogHandle 负责 (推荐)

     在source_location里，件名和函数名在编译时就已经确定，它们会存储在程序的只读数据段（常量区）。
     使用 const char* 只是保存一个指向内存地址的指针（8字节），完全没有任何内存分配（new/malloc）和字符串拷贝的开销

     全面拥抱 std::string_view：统的日志库（比如早期的 glog 或一些手写库）在这里会用 std::string。std::string 意味着什么？
     意味着每次打日志，都要在堆（Heap）上分配一次内存，把字符串拷贝过去。
     而 std::string_view 在底层只是一个 const char* 指针加上一个 size_t 长度：通常只有 16 字节
     它不拥有数据，只观察数据。这意味着构建一个 LogMsg 是一次零拷贝

三
加密模式选型：AES-GCM（推荐）
    输出格式：[12B nonce][ciphertext, 等长于明文][16B 认证 tag]
    唯一的坑：同一密钥下 nonce 绝不能重复。一旦重复，攻击者就能算出两条明文的 XOR、并能伪造任意密文
    但日志场景下这个坑很好避：每条日志一个单调递增的 counter 当 nonce（比如 12 字节 = 8 字节进程启动时戳 + 4
  字节序号），永远不会重。
  压缩选择了Zlib，但是要抽象出一层三方库无关的接口，可以定义一个公共的父类，不同的子类提供不同的压缩能力，后续如果有更好
  的压缩算法，也可以在新建一个子类
  同样的 加密模式也是 会抽象出一个加密接口，提供不同的加密算法实现。后续如果有更好的加密算法，也可以在新建一个子类
  目前选择了AES +ECDH

  回答：
  1. 1. Formatter 范围 → 只接 EffectiveFormatter（强类型）
  EffectiveSink 构造接 std::unique_ptr<EffectiveFormatter>，不是
  IFormatter。理由：管道下游是压缩+加密+审计解码，输入格式不固定就没法离线解析；强绑定可以让审计工具假定 proto 二进制单一格式。ConsoleSink
  那条线继续接 IFormatter，两条 sink 分开。
  2. mmap header 的 client_eph_pub → 每个 shard 文件头写一次
  同一进程一次 X25519 协商，整 shard 共用一把 AES key。文件头一次性 32B 足够审计工具还原；每块都写是纯浪费。
  3. 分片上限 + 淘汰总量 → 单片 64 MiB / 总量 2 GiB（都做成构造参数，给默认值）
  64 MiB 让 mmap 单次映射不夸张、压缩比稳定；2 GiB 适合中等规模服务。两个值都通过 MmapSinkConfig 暴露，调用方可改。
  4.分片文件命名 → log-YYYYMMDD-HHMMSS-mmm-NNNN.bin
  - YYYYMMDD-HHMMSS-mmm：开片时刻（毫秒，避免同秒多片冲突）
  - NNNN：同毫秒内的递增序号（保险）
  - 字典序 = 时间序，便于淘汰扫描和肉眼定位
  - .bin 表示二进制（区别于文本日志）
  5.淘汰定时器频率 → 30 秒一扫
  比每次 rotate 触发更宽松、不阻塞写入热路径；30s 对"总量缓慢逼近上限"的场景足够及时。实现上挂在 context 的 timer 接口（如果 [[2-a-context]] 没有
  timer 抽象就借一个独立 std::thread + condition_variable，沿用 ThreadPool 的 join 套路）。
  6. 内部日志 → fmt::print(stderr, "[logger-high][level] ...\n", ...)
  单文件 helper（utils/internal_log.h，分四级 info/warn/error/fatal）。绝对不调用 LogHandle，避免和 [[2-a-context]] 的递归保护冲突。fmt 已经是阶段 0
   依赖，零新增依赖。
  7.双 cache vs MmapStore 多 buffer → 复用既有 MmapStore 滚动 buffer，不另起一层
  [[2-b-mmap]] 里 MmapStore 已经是"一个 active buffer 接收写、其它在 seal/persist"，等价于 N-buffer 滚动。EffectiveSink 直接调
  MmapStore.Append(encrypted_frame)，不要自己再造一层 cache。"0.8 利用率触发"应该映射到 MmapStore 既有的 IsAcceptable 阈值（如果阈值不是 0.8，让
  MmapStore 参数化）。

    GCM 是 AES 的工作模式，不是另一个算法。
    AES 是分组密码，必须选一个模式：ECB / CBC / CTR / GCM /
  ...。你在开发记录"三"里写的就是
  "加密模式选型：AES-GCM"。所以完整名字是 AES-256-GCM。GCM 模式自带认证
  tag（防篡改），就是你记录里那段 [12B nonce][ciphertext][16B 认证 tag]
  的来源。
  为什么不用 fmt 文本压缩+加密？
   1. 可解码：加密后离线还要解出来，二进制 schema
  有版本可演进、跨语言、不会因为字段顺序变化解析挂掉
   2. 省字节：varint 编码比字符串紧凑，对压缩前体积有正面影响
  3. 明确边界：proto 帧自带长度，不像文本要靠分隔符

  mmap 用的是哪种方案？

  用的是你说的第二种（业务线程直接写 mmap 缓存 +
  切换），不是第一种（无锁队列 + 后台 memcpy）。但有一个补充。

   证据链：
  - MmapBuffer::Write —— 多生产者并发安全，用 fetch_add(offset_)
  抢偏移，业务线程直接 memcpy 到 mmap region
  - MmapStore::Append —— 调 master_->Write，写满返回 false →
  SwapMasterLocked_ 切到新 master，旧 buffer 投给 strand
  - MmapBuffer::Seal/Sync —— 后台 strand 上：Seal 等所有写者退出 →
  Sync() 调 msync → 接着 ForEachFrame 遍历 frame
  - MmapStore::DrainOne_ —— seal → sync → ForEachFrame(on_persist) →
  reset → 归还 free_
  本项目实际：后台 msync 之后 还要 ForEachFrame 把每条 frame 拷到最终 shard 文件
              mmap 文件是 循环复用的 ring buffer，shard 文件才是归档 

    为什么不让 mmap buffer 直接就是 shard？
  - buffer 要复用（reset 清零），shard 要长期保留并被 EvictionRunner 淘汰，生命周期不一样
  - buffer 内是 [len][payload] 帧格式，shard 要前置 48B header
  - 这样 shard 数量可控，不会满地都是小 mmap 文件
